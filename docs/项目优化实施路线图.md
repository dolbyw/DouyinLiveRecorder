# DouyinLiveRecorder 项目优化实施路线图

> 生成日期：2026-06-20
> 适用版本：当前仓库快照 `D:\Projects\DouyinLiveRecorder-main`
> 关联文档：`docs/技术架构与实现细节.md`

## 1. 文档目的

本文档用于将前期针对项目架构、代码实现、运行模式、依赖管理和性能瓶颈的分析，整理为一份可执行的改进计划。目标不是一次性重写整个项目，而是在尽量不破坏现有 40+ 平台录制能力的前提下，分阶段提升以下能力：

- 提升多直播间场景下的请求吞吐与整体稳定性；
- 降低 `main.py`、`spider.py` 等核心文件的维护复杂度；
- 建立统一的数据契约和模块边界，减少隐式耦合；
- 提升依赖、配置、日志和运行时的可观测性与可复现性；
- 为后续继续扩展平台或重构控制面打下基础。

## 2. 现状摘要

结合当前代码与技术报告，项目现阶段的主要特征如下：

- 主控制流程集中在 `main.py`，承担了配置读取、平台识别、线程管理、录制命令构建、转码、推送、备份等多重职责；
- 平台解析能力较强，但平台路由采用超长 `if/elif` 分发，扩展和维护成本高；
- 并发模型为“每个直播间一个线程 + 在线程内多次 `asyncio.run()` + FFmpeg 子进程”，结构简单但易产生隐藏状态问题；
- HTTP 请求热路径每次都会新建 `httpx.AsyncClient`，无法有效利用连接池；
- 请求层默认 `verify=False`，TLS 风险较高；
- 平台输出主要靠裸 `dict` 约定传递，缺少显式类型边界；
- 导入 `src` 时即触发 Node 检查，启动时再触发 FFmpeg 检查，导入副作用较强；
- 仓库缺少 lock 文件、测试体系和统一质量门禁；
- 敏感配置、日志和备份策略存在进一步治理空间。

## 3. 总体优化目标

本轮优化以“稳定性优先、渐进重构、保留兼容性”为基本原则，形成以下五个一级目标：

1. 收敛高风险问题
   优先解决 TLS、依赖漂移、错误处理、请求复用等直接影响稳定性和性能的问题。

2. 建立清晰模块边界
   将配置、平台适配、录制管线、网络层、运行时状态从超大主文件中逐步拆分。

3. 建立统一数据契约
   用显式模型替代散落的裸字典与字符串配置，减少隐式约定。

4. 升级控制面并发模型
   从线程包裹异步逐步过渡到统一事件循环和结构化并发。

5. 建立长期维护基础设施
   引入 lock 文件、lint、基础测试、运行指标和可观测能力。

## 4. 总体实施原则

### 4.1 分阶段替换，不做一次性重写

当前项目平台覆盖广，适配层价值很高。直接整体重写会导致高回归风险，因此不建议采取全量推翻方案，而应采用“小步提交、双轨兼容、逐步迁移”的模式。

### 4.2 先处理热路径，再处理结构美观

优先级上应先解决最影响性能与稳定性的部分，再处理目录结构和代码整洁度。当前最值得优先优化的热点包括：

- HTTP 客户端生命周期与连接池复用；
- 配置读取与依赖锁定；
- 平台分发与统一流信息契约；
- FFmpeg 命令构建与录制后处理；
- 并发状态管理与取消控制。

### 4.3 优先保持用户配置与运行方式兼容

改进过程中，原则上不立刻废弃现有 `config.ini`、`URL_config.ini`、命令行运行方式、FFmpeg 录制方式和主要输出目录结构。新结构应先提供兼容层，再择机推动迁移。

## 5. 目标架构蓝图

优化完成后的推荐架构如下：

```text
main.py
  -> 组装运行时
  -> 启动配置加载器、调度器、状态仓库

src/config_loader.py
  -> 读取 INI / 环境变量
  -> 输出强类型 Config 对象

src/http_clients/
  -> 复用 httpx.Client / AsyncClient
  -> 统一 timeout / retry / TLS / proxy / error

src/platforms/
  -> 平台适配器注册表
  -> 每个平台独立 match/fetch/normalize/select

src/models/
  -> Config / StreamInfo / RecordTask / Result / Enum

src/recorder/
  -> FFmpegBuilder
  -> RecorderProcess
  -> PostProcessor
  -> PathBuilder

src/runtime/
  -> Scheduler
  -> StateStore
  -> TaskGroup / 监控任务 / 停止控制
```

## 6. 分阶段优化路线

## 6.1 阶段一：热路径治理与基线收敛

### 目标

用最小改动解决当前项目最直接的性能、稳定性和供应链问题，为后续重构建立安全基线。

### 范围

- `src/http_clients/async_http.py`
- `src/http_clients/sync_http.py`
- `main.py`
- `pyproject.toml`
- 新增 lock 文件与开发工具配置

### 重点任务

1. 复用 HTTP 客户端
   - 将当前“每请求新建一个 `httpx.AsyncClient`”改为“少量长生命周期 client 池”；
   - 至少区分 `direct`、`proxy`、`abroad` 三类客户端；
   - 统一设置 timeout、headers、重试和关闭生命周期。

2. 收敛 TLS 策略
   - 默认开启证书校验；
   - 仅对个别平台保留可配置降级能力；
   - 不再默认全局 `verify=False`。

3. 统一错误模型
   - 请求异常不再简单转换为字符串返回；
   - 用显式异常或结果对象区分：连接失败、超时、HTTP 错误、空响应、JSON 解析失败等情况；
   - 首条错误信息必须保留原始根因。

4. 锁定依赖
   - 生成并提交 `uv.lock`；
   - 明确开发依赖组；
   - 尽量避免“仅声明最低版本但长期漂移”的状态。

5. 引入基础工程工具
   - 引入 `ruff` 作为统一 lint/format 工具；
   - 准备 `pytest` 基础环境。

### 建议新增文件

- `src/http_clients/client_pool.py`
- `src/http_clients/errors.py`
- `src/models/result.py`

### 阶段产出

- 请求层实现连接复用；
- 日志能准确标识失败阶段；
- 项目依赖可以复现安装；
- 为后续模块拆分建立统一基础设施。

## 6.2 阶段二：配置系统与统一数据契约

### 目标

把当前大量散落的字符串配置与裸字典流信息，收敛为强类型对象和统一契约。

### 范围

- `main.py`
- `src/utils.py`

### 重点任务

1. 定义配置模型
   - 建立 `AppConfig`、`PushConfig`、`CookieConfig`、`AccountConfig` 等对象；

   - 建立 `StreamInfo`、`PlatformResult`、`RecordRequest` 等对象；

   - 继续兼容 `config.ini`、`URL_config.ini`；

- 如果更偏向性能，可在模型层使用 `msgspec.Struct`；

其中：

- `fetch()` 负责调用现有 `spider.py`；
- `normalize()` 负责对接阶段二已经形成的统一边界；
- `select_stream()` 负责调用现有 `stream.py` 或直接组装最终流结果；
- 原型期不强制立刻加入更复杂的 `build_record_headers()` 或平台专属上下文对象。

   - `select_stream()` 复用现有 `stream.get_douyin_stream_url()`。

3. 为 B站实现样板适配器

   - 原型期不删除旧逻辑；
   - 新旧两条路径同时存在；
   - 先把“注册表路径可跑通”作为目标，而不是立刻追求大面积删除旧代码。

目标：

设计基线：

- 采用“单循环控制面 + 有界同步执行面”，平台探测与轮询运行在单一事件循环，阻塞录制管线运行在专用执行器；
- 房间级异常在 supervisor 边界隔离，不允许一个直播间故障取消全部房间；
- 详细设计见 `docs/superpowers/specs/2026-06-20-pr5-async-runtime-design.md`。

首批改造文件：

- `src/runtime/app.py`
- `src/runtime/state.py`

- `src/runtime/recording.py`
- `main.py`
- `src/http_clients/async_http.py`
- `src/platforms/dispatch.py`
- `src/recorder/pipeline.py`

建议新增结构：

- `src/runtime/app.py`
  - 提供统一启动入口，负责创建事件循环、状态仓库和调度器；
- `src/runtime/state.py`
  - 定义 `StateStore`，通过领域接口维护配置期望状态与运行状态，并提供不可变快照；
- `src/runtime/scheduler.py`
  - 负责房间任务创建、取消、配置协调和房间级故障隔离；
- `src/runtime/limiter.py`
  - 提供取消安全、支持运行时扩缩容的网络请求限流器；
- `src/runtime/recording.py`
  - 将同步 `RecordingPipeline` 提交到专用执行器，并桥接停止令牌与状态回调；
- `src/runtime/models.py`
  - 定义监控任务、并发配置和运行期快照等最小运行时模型。

最小边界设计：

1. `StateStore`
   - 提供 `replace_desired_rooms`、`mark_recording_started/finished`、`request_stop` 和 `snapshot` 等领域接口；
   - 不允许主流程继续直接操作散落的全局集合；
   - 不保存 asyncio 任务句柄；任务所有权归调度器。

2. `RuntimeScheduler`
   - 接收房间监控任务并统一调度；
   - 使用真实生效的限流器替代“只改显示值、不改执行体”的并发控制；
   - 房间业务异常只影响当前房间，应用基础任务异常才触发整体关闭；
   - 明确处理任务取消、退避节奏和配置增删。

3. `RuntimeApp`
   - 在单一事件循环里启动平台探测和控制任务；
   - 录制管线通过专用执行器运行，不得阻塞事件循环；
   - 对旧线程入口提供兼容包装，避免一次性重写全部启动方式。

建议拆分与落地步骤：

1. `PR5A`：状态仓库
   - 抽出 `StateStore`、任务描述对象和并发配置对象；
   - 梳理并替换 `main.py` 的共享状态读写，保留薄兼容门面。

2. `PR5B`：异步调度器
   - 升级 Python 3.11，建立单一事件循环、房间 supervisor 和动态限流器；
   - 迁移平台探测热路径，移除其中的 `run_async()` 和每房间线程。

3. `PR5C`：录制桥接与应用生命周期
   - 使用专用执行器运行同步录制管线，以线程安全停止令牌连接 async 控制面与 FFmpeg；
   - 实现 SIGINT、SIGTERM、配置删除和 URL 注释的有序停止；
   - 兼容壳可保留一个版本周期，但不得继续创建每房间线程。

## 13. 结论

DouyinLiveRecorder 当前最有价值的资产是成熟的平台适配层和实用的常驻录制能力；当前最大的限制则是请求热路径、超大主控文件、隐式数据契约和混合并发模型。

因此，本项目最合理的演进方式不是激进重写，而是：

`先治理热路径与依赖 -> 再建立契约与适配层 -> 再升级调度与运行时`

按照本文档路线执行，可以在保留现有业务能力的前提下，逐步把项目演进为一个：

- 更稳定；
- 更高性能；
- 更易维护；
- 更适合长期扩展；

的多平台直播录制系统。
