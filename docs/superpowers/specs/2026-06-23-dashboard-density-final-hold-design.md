# 高密度监控面板与转码完成驻留设计

## 目标

在不降低直播间列表可读性的前提下，提高主面板有效信息密度，并让第一次 Ctrl+C 的完整收尾结果停留在屏幕上，直到用户明确退出。

本次只修改展示状态、运行时详情采集和优雅退出后的驻留行为，不改变录制、分段或转码参数。

## 已确认布局：方案 A

### 标题与时间

标题行居中显示 `DouyinLiveRecorder  ● 运行正常`。运行状态不再单独占据左侧区域。

标题下方保留一行元数据：

- 左侧：`当前  YYYY-MM-DD HH:MM:SS`
- 中间：`启动  YYYY-MM-DD HH:MM:SS`
- 右侧：`FFmpeg 正常` 或 `FFmpeg 异常`

两个时间都使用本地时区并精确到秒。程序启动时间在进程生命周期内保持不变，当前时间每次刷新更新。

### 摘要与配置

摘要仍为五个等宽卡片，但删除与标题区域重复的 FFmpeg 卡，改为：

1. 直播间
2. 录制中
3. 监控中
4. 转码中
5. 错误

配置区前两行继续显示格式、质量、分段、监控间隔、并发、代理、磁盘和自动转码状态。保存路径独占第三行，使用配置中的绝对路径；只在终端实际宽度仍不足时从左侧截断，并优先完整保留最后两级目录。

### 实时详情列

“详情”标题改为“实时详情”，并按房间状态显示不同信息：

- `等待`：`等待首次检测`
- `监控`：`下次检测 MM:SS`
- `录制`：`文件大小 · 平均码率`，例如 `128.4 MB · 4.7 Mbps`
- `转码`：`文件序号/总数 · 文件名`，例如 `2/3 · 主播_001.ts`
- `重试`：精简后的最新错误
- `停用`：`配置已停用`

录制大小来自当前输出文件。分段录制使用输出通配模式统计已生成分段的总大小；平均码率按总字节数和录制时长计算。文件在刷新瞬间不存在、被轮转或无法读取时，详情退化为当前文件名，不产生面板错误。

监控倒计时使用“最近一次真实探测完成时间”，不能用界面刷新时间伪造。异步监控在 `RoomMonitor.run_once` 成功结束后更新该时间；兼容监控在离线或录制周期探测完成时更新。倒计时最小显示 `00:00`，下一次探测完成后重新从配置间隔计时。

## 详情为空的根因

当前渲染器只读取 `room.task_name or room.last_error`。`mark_recording` 会主动清空 `task_name`，监控状态也从未设置任务名称，因此绝大多数房间的详情必然为空。这不是 Rich 列宽问题，而是状态模型没有采集并传递对应数据。

修复应扩展统一的 `DashboardRoom` 快照字段，而不是在渲染器中猜测运行状态：

- `output_path`：当前录制输出路径或分段路径模式
- `last_checked_at`：最近一次真实探测完成时间
- 现有 `task_name`：继续承载转码文件序号和名称

渲染器只执行轻量、本地、容错的文件大小读取和格式化，不进行网络请求。

## 优雅退出驻留状态机

应用展示状态新增四个阶段：

1. `运行正常`：常规监控和录制。
2. `正在停止`：第一次 Ctrl+C 已收到，禁止新探测和新录制，正在停止 FFmpeg。
3. `正在收尾`：至少一个录制文件正在转码或后处理。
4. `收尾完成`：所有活动录制和后处理结束，程序等待用户按键。

第一次 Ctrl+C 仍沿用现有安全流程：请求停止、等待录制进程、完成全部本次 TS→MP4。录制集合清空后不再立即 `sys.exit(0)`，而是：

1. 将面板状态设置为 `收尾完成`。
2. 写入一条不可重复的最终事件，概括转码成功数和失败数；已有逐文件事件继续保留。
3. 在面板底部显示 `按任意键退出  |  Ctrl+C 强制退出`。
4. 阻塞等待一个控制台按键；任意键返回退出码 0。

等待期间面板继续每秒刷新当前时间，但不再刷新配置或发起任何探测。第二次 Ctrl+C 沿用 `ShutdownControl` 的强制退出路径，退出码仍为 130。

Windows 使用 `msvcrt.getwch()` 读取单个按键，不要求 Enter。非交互标准输入或非 Windows 环境使用可测试的输入等待适配器；如果没有可读控制台，记录最终摘要后正常退出，禁止永久挂死后台任务。

## 数据与组件边界

### `src/dashboard_state.py`

负责保存不可变快照所需的启动时间、应用阶段、输出路径、真实探测时间和最终提示。状态更新保持线程安全。应用阶段只能按 `运行 → 停止 → 收尾 → 完成` 前进，较早阶段不能覆盖较晚阶段。

### `main.py`

在录制开始时传入真实输出路径，在真实探测完成回调中更新时间。信号处理器只编排停机、标记完成和等待按键，不负责拼装 Rich 文本。

### `src/cli_ui.py`

根据快照呈现方案 A。路径、文件大小、码率、倒计时和最终提示均由独立格式化函数生成；任何文件系统异常只影响对应详情单元格。

## 错误处理

- 文件统计失败：显示当前文件名或 `正在写入`，不增加全局错误数。
- 探测时间未知：显示 `等待首次检测`。
- 转码失败：保留 TS，最终事件显示失败数量，仍进入 `收尾完成` 等待用户查看。
- 等待按键时再次 Ctrl+C：立即强制退出。
- 无交互控制台：不等待按键，输出最终日志后退出。
- Rich 渲染降级：纯文本界面同样显示当前/启动时间、实时详情和最终退出提示。

## 测试与验收

自动化测试覆盖：

1. 标题包含运行状态，标题下显示两个完整日期时间，且 FFmpeg 不再作为摘要卡。
2. 第五张摘要卡为错误数。
3. 保存路径在宽终端独占整行并保留更多目录层级；窄终端安全截断。
4. 监控倒计时只由真实探测时间驱动，不随面板刷新重置。
5. 非分段和分段录制都能显示总文件大小与平均码率。
6. 转码显示文件序号、总数和文件名。
7. 文件缺失或统计异常不会导致面板退出。
8. 第一次 Ctrl+C 完成转码后进入 `收尾完成`，不会立即调用 `sys.exit`。
9. 任意键触发正常退出；等待期间第二次 Ctrl+C 触发退出码 130。
10. 非交互输入不会永久等待。
11. 全量 pytest、真实 FFmpeg TS→MP4 冒烟和 PyInstaller 打包继续通过。

人工验收时启动至少一个监控房间和一个录制房间，确认详情每秒变化。按一次 Ctrl+C 后等待转码完成，确认最终界面持续保留至少一分钟，随后按普通按键退出；另一次运行验证完成界面上的 Ctrl+C 可以立即退出。

## 非目标

- 不显示或泄露直播源 URL、Cookie、Token 或账号信息。
- 不加入键盘菜单、房间选择或交互式控制功能。
- 不扫描历史 TS，也不修改现有转码成功判定。
