# PR5 统一异步运行时设计

## 目标

将当前“每个直播间一个线程、线程内反复 `asyncio.run()`、共享全局集合”的控制面迁移为单一事件循环驱动的运行时，同时保持平台探测、录制格式、配置文件和输出行为兼容。

PR5 分三批交付：PR5A 建立领域状态仓库，PR5B 建立异步房间调度和动态网络限流，PR5C 将同步录制执行面安全接入并完成优雅退出。每一批都必须形成可独立测试、可回退的工作状态。

## 兼容性决策

项目最低 Python 版本从 3.10 升级到 3.11，以直接使用 `asyncio.TaskGroup`，不引入结构化并发兼容库。打包配置、Ruff 目标和文档必须同步更新。

`config.ini`、`URL_config.ini`、命令行启动方式、平台适配器接口、`RecordingPipeline` 的同步公共接口以及用户可见输出目录均保持兼容。

## 范围

本次包含：

- 新增 `src/runtime/`，提供运行时模型、状态仓库、动态限流器、房间调度器和应用生命周期；
- 将平台探测与轮询迁移到单一事件循环，消除热路径中的 `run_async()`；
- 将阻塞录制管线放入专用执行器，不阻塞事件循环；
- 使用线程安全停止令牌连接 async 控制面与同步录制执行面；
- 将 `recording`、`running_list`、`url_comments` 等运行态迁移到领域接口；
- 明确房间级故障隔离、重试退避、配置增删和应用级关闭顺序；
- 让运行中的并发配置变化真实影响后续网络请求。

本次不包含：

- 重写平台适配器、FFmpeg 管线或后处理业务规则；
- 将 FFmpeg 子进程本身改写为原生异步子进程；
- 新增 GUI、Web API、数据库或分布式调度；
- 自动重试单次平台 HTTP 请求；重试仍发生在房间轮询层；
- 将推送、备份等所有外围线程一次性异步化。

## 方案选择

采用“单循环控制面 + 有界同步执行面”。平台探测、房间轮询、配置协调、状态快照和限流由一个 asyncio 事件循环拥有；长时间阻塞的录制管线在专用线程池中执行，通过停止令牌和线程安全状态投递与控制面通信。

不采用一步到位的全 async 重写，因为 `RecordingPipeline`、直连下载和大量平台逻辑已经稳定；也不保留“每房间一个线程”的主模型，因为这无法复用异步 HTTP client，并继续放大共享状态竞争。

## 模块架构

- `src/runtime/models.py`：定义不可变的 `RoomSpec`、`RoomStatus`、`RuntimeSnapshot` 和关闭原因；
- `src/runtime/state.py`：定义由事件循环单线程拥有的 `StateStore`，只暴露领域操作和不可变快照；
- `src/runtime/limiter.py`：定义取消安全的 `AdjustableLimiter`；
- `src/runtime/scheduler.py`：维护 URL 到房间任务的映射，负责增删、故障隔离和停止；
- `src/runtime/app.py`：创建共享依赖、协调配置、启动调度器并执行有序关闭；
- `src/runtime/recording.py`：把同步 `RecordingPipeline.run()` 提交到专用执行器，并桥接停止与回调；
- `src/runtime/__init__.py`：导出稳定公共接口。

任务句柄只归 `RuntimeScheduler` 所有，不进入公开状态仓库。状态仓库描述业务状态，调度器描述执行机制。

## 状态模型

`StateStore` 由运行时事件循环独占写入，不提供通用 `get/set/update`。第一批领域接口为：

- `replace_desired_rooms()`：原子替换配置期望，并返回新增、删除和修改集合；
- `mark_monitoring()`：记录房间进入监控；
- `mark_recording_started()` 与 `mark_recording_finished()`：维护录制名、质量和开始时间；
- `mark_room_error()`：维护房间错误信息和连续失败次数；
- `request_room_stop()` 与 `request_shutdown()`：记录停止意图；
- `snapshot()`：返回不可变运行时快照，供显示线程或未来控制面读取。

房间以规范化 URL 作为稳定标识。配置状态、观测状态和调度任务分离，避免 `url_comments` 同时承担配置、停止信号和运行状态三种职责。

同步录制线程不得直接修改仓库。它通过 `loop.call_soon_threadsafe()` 投递录制开始、结束和错误事件。兼容期内旧显示逻辑只读取快照。

## 动态网络限流

`AdjustableLimiter` 限制并发平台网络探测，不限制房间任务总数，也不限制已经运行的 FFmpeg 进程。

它提供 `acquire()`、`release()`、异步上下文管理和 `set_limit()`：

- 扩容立即唤醒等待任务；
- 缩容不取消已经获得许可的任务，只阻止新的任务进入，直到活动数低于新上限；
- 等待任务取消后必须从等待队列移除，不消耗许可；
- 上限必须大于零；
- 维护 `active_count`、`waiting_count` 和 `limit` 只读指标。

动态错误窗口只负责计算目标并发值，实际执行约束完全由该限流器承担。

## 结构化并发与故障隔离

`RuntimeApp` 使用应用级 `TaskGroup` 管理配置协调、状态显示和调度器生命周期。应用级基础任务异常可触发整体关闭。

每个房间由独立 supervisor 协程管理。平台探测异常在房间边界被捕获、记录并退避，不允许逃逸到应用级 `TaskGroup` 后取消其他房间。只有 `CancelledError` 必须原样传播。

房间内部按“限流探测 → 根据状态录制或等待 → 可取消延时”循环。配置删除、注释或应用退出会取消房间 supervisor；修改质量或名称采用取消旧任务并以新 `RoomSpec` 重建，避免任务读取一半新旧配置。

## 阻塞录制边界

`RecordingPipeline.run()` 保持同步，由 `RecordingExecutor` 提交给专用 `ThreadPoolExecutor`。事件循环不得直接调用同步管线。

每次录制持有线程安全的 `threading.Event`：房间被删除时设置房间停止事件，应用退出时设置全局停止事件。管线已有的 `should_comment_stop` 和 `should_exit` 回调读取这些事件，使取消 async 等待的同时能够终止底层 FFmpeg。

取消等待录制结果并不等于线程已停止。调度器必须先设置停止事件，再等待执行器 future 在带超时的关闭阶段结束。超时后记录残留任务并继续关闭，不允许事件循环无限等待。

## HTTP 生命周期

迁移后的平台探测直接 `await try_resolve()` 或平台异步函数，不再从事件循环中调用 `run_async()`。共享 async client 绑定唯一事件循环，在 `RuntimeApp` 关闭阶段统一调用 `close_async_clients_for_current_loop()`。

`run_async()` 仅保留给尚未迁移的同步入口，运行时模块不得调用它。测试需要静态验证该边界。

## 有序关闭

应用关闭顺序固定为：

1. 停止接收配置变更并设置全局关闭意图；
2. 设置所有房间停止事件；
3. 取消轮询、限流等待和延时任务；
4. 等待录制执行器中的 FFmpeg 管线退出；
5. 关闭共享 async HTTP clients；
6. 关闭专用执行器；
7. 生成最终状态快照并结束事件循环。

SIGINT 与 SIGTERM 转换为运行时关闭请求，不在信号处理器中直接 `sys.exit()`。重复关闭必须幂等。

## 分批迁移

### PR5A：状态仓库

先新增模型与 `StateStore`，补齐状态转换和快照测试。通过薄兼容门面把现有状态读写逐步切到仓库，线程调度暂时保持不变。此阶段不得把 asyncio 对象放入仓库。

### PR5B：异步调度器

升级 Python 3.11，新增 `AdjustableLimiter`、房间 supervisor 和 `RuntimeScheduler`。先使用假的探测与录制函数跑通增删、异常隔离、取消和动态限流，再接入平台异步分发。完成后房间线程和探测热路径中的 `run_async()` 被移除。

### PR5C：录制桥接与应用生命周期

新增 `RecordingExecutor`，把同步录制管线放入专用执行器；接入线程安全状态事件和停止令牌；最后由 `RuntimeApp` 替换主循环入口并实现完整关闭。旧入口保留为一个版本周期内可切换的兼容壳，但不得继续创建每房间线程。

## 测试策略

严格使用测试驱动开发，不依赖公网：

- `StateStore`：配置差异、重复操作、录制状态、停止状态和不可变快照；
- `AdjustableLimiter`：初始上限、扩容、缩容、等待取消、许可归还和公平唤醒；
- `RuntimeScheduler`：房间增删、配置修改、重复协调、单房间异常隔离、取消传播；
- `RecordingExecutor`：事件循环不被阻塞、开始/结束事件线程安全投递、房间停止、全局停止和关闭超时；
- `RuntimeApp`：启动顺序、幂等关闭、HTTP client 关闭和异常清理；
- 主流程接线：运行时路径不调用 `run_async()`，不再创建每房间线程；
- 回归：完整 pytest、Ruff、格式检查和 `py_compile`。

人工长稳测试至少运行 2 小时，记录起止线程数、asyncio 任务数、RSS 内存、HTTP client 数量和活动 FFmpeg 数量。验收要求空闲期任务与线程不随轮询次数持续增长，配置删除后对应任务与录制进程能退出。

## 验收标准

- 项目最低 Python 版本和工具配置统一为 3.11；
- 所有房间平台探测运行在同一事件循环，热路径无 `asyncio.run()`；
- 单房间未捕获业务异常不会终止其他房间；
- 动态限流扩容和缩容均由并发行为测试证明真实生效；
- 同步录制不阻塞事件循环，取消会传递到底层 FFmpeg 停止条件；
- 运行状态通过领域接口维护，主流程不直接修改旧全局集合；
- SIGINT、SIGTERM、配置删除和 URL 注释都能触发对应作用域的有序停止；
- 自动化测试和静态检查通过，人工长稳指标无单调增长；
- 每一批均可独立回退，不改变平台解析、录制格式和输出目录行为。
