# 主界面精修与 Ctrl+C 完整转码设计

## 目标与范围

本次改动包含两个边界清晰的目标：

1. 在不降低当前信息密度的前提下，将 Rich 主界面精确调整为已确认的「布局 A」。
2. 修复 Windows 下按 Ctrl+C 后 FFmpeg 返回 255 被误判为录制失败、当前 TS 未转码且录制短暂重启的问题。

只保证本次改动之后产生的录制在 Ctrl+C 退出时完整转码；不扫描或补转历史遗留 TS。

## 已确认的界面契约

宽终端界面采用固定的五级信息层次：

1. 标题与健康状态：`DouyinLiveRecorder` 居中，显示当前时间、绿色运行状态与 FFmpeg 健康状态。
2. 五项摘要卡：直播间、录制中、监控中、转码中、FFmpeg。每项由醒目的值和弱化标签组成，并保持等宽排列。
3. 两行配置网格：格式转换关系、质量、分段、间隔、并发、代理、磁盘和保存目录，按四列排列。TS 且开启自动转码时显示为 `TS → MP4`；分段时间使用易读的分钟表达；保存路径优先保留末级目录，超长部分从左侧省略。
4. 主表：列顺序固定为序号、直播间、状态、质量、时长/进度、详情。直播间列获得最大弹性宽度，状态和质量保持紧凑，时长/进度不被详情列挤压。录制详情显示当前文件体积（可获得时），转码详情显示文件名，监控详情可显示下次检测倒计时。
5. 双列事件区：最近事件最多四条，按两列排列；无错误时显示绿色健康提示，有错误时显示最新精简错误。

状态视觉遵循确认稿：录制为绿色，转码为琥珀色，监控为青色，错误为红色，次要文本为灰色。状态包含简洁符号，使用户不只依赖颜色识别状态。

窄终端仍使用同一快照数据。宽度不足时依次降级为隐藏详情列、摘要卡换行、配置压缩为核心字段；直播间、状态、质量、时长/进度始终保留。

## 问题证据与根因

日志在 Ctrl+C 后依次出现：

- 停止并转码提示；
- 三个 FFmpeg 录制进程返回 255 并被记录为录制错误；
- 同一直播间再次进入「正在直播中」；
- 原录制 TS 保留，同时生成以退出后时间命名的数百 KB MP4。

`RecorderProcess.run` 当前先调用 `process.poll()`。如果 FFmpeg 已响应 `q` 并以 255 退出，轮询会立即把非零返回码归类为 `FAILED`，而没有再次检查 `should_exit()`。`PostProcessor` 只处理成功结束原因，因此原 TS 被跳过。失败分支又允许调度链进入后续检测或录制，形成退出期间的短暂重启。

## Ctrl+C 停机与转码状态机

第一次 Ctrl+C 执行以下顺序：

1. `ShutdownControl` 只接受一次优雅退出请求，并设置全局退出意图。
2. 异步运行时和兼容线程收到停止令牌；调度入口从此拒绝启动新的探测或录制。
3. 活跃 FFmpeg 录制进程在 Windows 下通过标准输入收到 `q`。
4. 无论代码先观察到停止意图还是先观察到 FFmpeg 的非零退出码，只要停止意图已经成立，该进程都归类为 `EXIT_STOPPED`，而不是 `FAILED`。
5. `EXIT_STOPPED` 进入正常后处理。当前录制产生的全部 TS 按既有顺序逐个转为 MP4，并持续更新面板进度。
6. 转码先写入 `.converting.mp4`，成功后原子替换最终 MP4；只有最终文件提交成功，才按配置删除源 TS。
7. 所有录制和转码任务结束后主进程退出。

第二次 Ctrl+C 保持现有行为，立即以退出码 130 强制终止。强制终止不承诺完成转码，但原子转换规则保证未完成时仍保留源 TS。

## 组件改动边界

### `src/recorder/process.py`

集中负责从进程状态和停止意图推导 `EndReason`。自然发生的非零退出仍是 `FAILED`；只有已经请求的房间停止或程序退出才能映射为对应的正常停止原因。此处修复 poll/stop 的竞态，不在上层硬编码“255 永远成功”。

### `main.py`

在进入直播探测、开始新录制和兼容调度创建线程之前检查停机意图。退出中的正常停止不增加错误计数、不进入重试状态，也不再开始下一轮。当前转换回调继续驱动同一面板，不创建额外输出界面。

### `src/cli_ui.py`

只修改快照的视觉呈现和响应式布局，不把新的业务状态塞进渲染器。摘要计数从现有房间状态计算；无法从当前快照可靠获得的文件体积或倒计时不伪造，详情保持为空或显示现有任务名。

## 错误处理

- 自然发生的 FFmpeg 255 仍报告为失败，避免掩盖真实录制故障。
- 转码失败时记录具体错误，保留源 TS，并在面板显示精简错误。
- 某个文件转码失败不删除该源文件；其余文件按既有逐文件策略继续处理并分别报告结果。
- 停机后禁止新任务，但已进入后处理的任务不因第一次 Ctrl+C 被取消。
- 历史 TS 不参与启动扫描或自动补转。

## 测试设计

自动化测试按红—绿顺序覆盖：

1. Windows 进程在退出意图已设置、`poll()` 先返回 255 时，结果为 `EXIT_STOPPED`。
2. 没有停止意图时返回 255，结果仍为 `FAILED`。
3. Ctrl+C 正常停止产生的 TS 会进入 `PostProcessor` 转码。
4. 停机意图成立后，不再执行新的探测、录制或兼容线程调度。
5. 正常停机不记录“直播录制出错: 返回码 255”，也不增加错误计数。
6. 宽终端渲染包含五项摘要、`TS → MP4`、两行四列配置、六列主表和双列事件区。
7. 状态符号、颜色语义和关键文案与确认稿一致。
8. 窄终端保留直播间、状态、质量、时长/进度四个核心列。
9. 现有 recorder、runtime、dashboard 与完整 pytest 测试集继续通过。

人工验收使用至少两个正在直播的房间：等待录制稳定后按一次 Ctrl+C，确认不再出现新的「正在直播中」，原 TS 逐个显示转码进度，最终 MP4 可被 ffprobe 读取，且在配置允许删除源文件时不残留 TS。界面按已确认的布局 A 与浏览器对比稿逐项核对。

## 非目标

- 不补转已有历史 TS。
- 不改变编码质量、封装参数或分段策略。
- 不重构平台解析和直播源获取逻辑。
- 不增加新的图形界面或交互命令。
