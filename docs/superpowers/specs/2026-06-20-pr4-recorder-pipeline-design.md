# PR4 录制管线模块化设计

## 目标

将路径与命名、FFmpeg 命令构建、录制进程生命周期、直连下载和录制后处理从 `main.py` 拆分到可独立测试的 `src/recorder/` 包，同时保持现有配置格式、运行入口、输出目录结构和直播平台探测行为兼容。

本次完整修复以下已知缺陷：

- 猫耳 FM、Look 以及显式音频格式录制完成后继续落入通用视频分支；
- 分段 FLV 后处理把含 `%03d` 的输出模板当作真实文件；
- 分段 M4A 使用 MPEG-TS 容器但写入 `.m4a` 扩展名；
- MP4 转换在不同结束路径和格式分支中的触发条件不一致。

## 范围

本次包含：

- 新增录制请求、输出计划、进程结果和管线结果等强类型数据对象；
- 集中实现保存目录、标题与主播名清洗、时间目录、分段模板和扩展名策略；
- 集中实现 TS、FLV、MKV、MP4、MP3、M4A 的分段与非分段 FFmpeg 参数；
- 抽离 FFmpeg 进程启动、轮询、优雅停止、等待和结束原因判断；
- 抽离 TS 转 MP4、分段文件发现、M4A 兼容处理和自定义脚本触发；
- 将直连 FLV 下载作为显式录制策略接入管线；
- 在 `main.py` 中接入新管线并删除被替代的命令拼接和重复格式控制流；
- 补充模块测试、缺陷回归测试和主流程接线测试；
- 更新项目优化路线图中的 PR4 状态。

本次不包含：

- 改造平台注册表、直播探测或选流逻辑；
- 重构线程调度、全局状态仓库或统一事件循环，这些属于 PR5；
- 更改 `config.ini`、`URL_config.ini` 或命令行入口；
- 强制进行依赖真实直播间、开播状态或公网质量的自动化测试；
- 改变 Shopee、花椒等直连下载平台既有的不分段、不转码、不执行完成脚本行为。

## 方案选择

采用“兼容门面式完整拆分”。新模块使用显式数据对象和依赖注入实现可测试边界，`main.py` 继续拥有平台探测、推送、错误窗口和现有全局录制状态，通过请求对象与回调接入录制管线。

该方案比彻底重写为独立录制服务更少侵入 PR5，也比机械搬移函数更能消除隐式全局依赖和格式分支缺陷。

## 模块架构

新增 `src/recorder/`：

- `models.py`：定义不可变的 `RecordRequest`、`OutputPlan`、`ProcessResult`、`PipelineResult`、结束原因枚举及后处理结果；
- `pathing.py`：负责名称清洗、输出目录、文件名、扩展名、分段模板和用于发现实际分段文件的 glob；
- `ffmpeg_builder.py`：负责公共输入参数、代理、headers、国内/海外参数以及六种输出格式策略；
- `process.py`：负责子进程启动、轮询、跨平台优雅停止、等待和返回码处理；
- `postprocess.py`：负责 TS 转 MP4、分段文件解析、M4A 兼容处理和自定义脚本执行；
- `pipeline.py`：选择 FFmpeg 或直连下载策略，串联路径、构建、执行和后处理；
- `__init__.py`：导出稳定公共接口。

模块不得直接导入 `main.py` 或读取其全局变量。时间、操作系统、停止条件、进程创建器、直连下载器和后处理执行器通过请求字段或构造参数传入。

## 数据模型

`RecordRequest` 至少携带：

- 直播间标识：主播名、平台名、直播间 URL、媒体源 URL、标题；
- 输出设置：根目录、保存格式、按主播/日期/标题建目录、标题写入文件名、是否分段及分段秒数；
- 网络设置：代理、headers、是否海外参数；
- 录制设置：仅音频、直连 FLV、强制 HTTPS、转 MP4、转 H.264；
- 生命周期输入：是否被注释、是否全局退出的可调用判断；
- 兼容回调：进度、日志、录制状态和自定义脚本所需的窄接口。

`OutputPlan` 同时保存模板路径和实际文件匹配规则。非分段输出的实际路径等于模板路径；分段输出使用 `%03d` 模板，并由独立 glob 查找真实文件。

`ProcessResult` 明确区分：

- `COMPLETED`：FFmpeg 自然正常结束；
- `COMMENT_STOPPED`：URL 被注释后优雅停止；
- `EXIT_STOPPED`：程序退出后优雅停止；
- `FAILED_TO_START`：子进程未成功启动；
- `FAILED`：子进程异常退出。

结果保留返回码和原始异常，避免用布尔值混淆结束原因。

## 路径与命名

路径策略保持现有用户可见结构：根目录下按平台，并按配置选择主播、日期和标题层级。标题与主播名沿用现有字符清洗语义；清洗为空时使用 `空白昵称`。

文件名由主播名、可选标题、`YYYY-MM-DD_HH-MM-SS`、可选 `_%03d` 分段标记和格式扩展名组成。

扩展名固定映射为：TS → `.ts`、FLV → `.flv`、MKV → `.mkv`、MP4 → `.mp4`、MP3 → `.mp3`、M4A → `.m4a`。

目录创建由路径模块完成；命令构建器只消费已经确定的输出计划。

## FFmpeg 命令构建

公共输入参数保持现有配置，包括 user agent、协议白名单、探测时长、缓冲、重连、时间戳修正、代理和平台 headers。海外参数继续使用更大的 timeout、probe、buffer 和 mux queue 值。

输出格式矩阵：

| 格式 | 非分段 | 分段 |
|---|---|---|
| TS | 音视频 copy，MPEG-TS | 音视频 copy，segment + MPEG-TS |
| FLV | 音视频 copy，AAC bitstream filter | 音视频 copy，segment + FLV |
| MKV | 音视频 copy，Matroska | 视频 copy、音频 AAC，segment + Matroska |
| MP4 | 音视频 copy、faststart | 视频 copy、音频 AAC，fragmented MP4 segment |
| MP3 | 仅音频、libmp3lame 320k | 同编码并使用 segment muxer |
| M4A | 仅音频、AAC 320k、iPod/MP4 容器 | AAC 320k，segment muxer 显式使用与 `.m4a` 一致的 iPod/MP4 容器 |

仅音频平台无论全局视频格式为何，都生成一个音频输出计划并在该计划结束后结束本轮管线。显式 MP3/M4A 配置遵循同一规则。

H.265 FLV 的现有 TS 回退保留在请求组装或格式解析边界，命令构建器接收最终格式，不自行检查平台字典。

## 进程生命周期

`RecorderProcess` 使用注入的进程工厂启动 FFmpeg，并按固定间隔轮询：

1. 进程已退出时读取返回码并分类；
2. URL 被注释时发起优雅停止并等待；
3. 程序退出标志出现时发起优雅停止并等待；
4. 否则继续轮询并报告可选进度。

Windows 通过 stdin 写入 `q` 并刷新；Linux/macOS 发送 `SIGINT`。停止请求只发送一次。等待超时和停止发送异常保留为失败信息，不允许无限阻塞。

状态清理由调用方兼容回调和 `pipeline.py` 的 `finally` 共同保证。进程模块不直接操作全局 `recording`、`running_list` 或配置文件。

## 直连 FLV 策略

Shopee、花椒等强制直连平台由 `RecordingPipeline` 在构建 FFmpeg 命令前选择直连策略。直连下载器通过注入接口复用现有实现。

直连策略保持现状：只写单个 FLV 文件，不分段、不转 MP4、不执行完成脚本。其结果仍统一包装为 `PipelineResult`，并保证状态清理。

## 后处理

只有 `COMPLETED`、`COMMENT_STOPPED` 和 `EXIT_STOPPED` 可进入成功型后处理；`FAILED_TO_START` 与 `FAILED` 不触发转码或完成脚本。

优雅停止与自然结束使用相同后处理规则：

- TS 在启用转换时转为 MP4；
- 分段输出通过 `OutputPlan` 的 glob 按顺序枚举实际文件，逐个处理；
- FLV 分段不得把 `%03d` 模板传给文件存在检查或转码函数；
- M4A 直接生成正确容器，不再依赖录制后修正错误容器；兼容处理函数仅用于明确识别出的历史文件；
- 自定义脚本只在录制成功且策略允许时执行一次。

后处理步骤独立返回成功或失败。后处理失败不得删除可用原文件，也不得把成功录制改写成进程失败；`PipelineResult` 同时报告录制结果和后处理错误。

## 主流程接线

`main.py` 继续负责：

- 选取 `real_url`；
- 根据平台和配置确定最终保存格式、仅音频与直连策略；
- 维护推送、错误窗口和用户可见状态；
- 创建 `RecordRequest` 并调用 `RecordingPipeline.run()`；
- 根据 `PipelineResult` 更新日志、错误计数和下一轮轮询。

新接线完成后，`main.py` 不再包含六种格式的 FFmpeg 参数拼接，也不再直接实现 FFmpeg 子进程轮询与格式后处理。现有通用辅助函数可在迁移期间作为薄兼容包装，但不得保留第二套行为实现。

## 错误处理

- 不支持的格式、无效分段时长或缺失媒体源在启动前失败；
- 路径创建失败返回路径阶段错误；
- 进程启动异常保留原始异常；
- 非零返回码包含结束原因和返回码；
- 优雅停止失败受等待超时保护；
- 后处理错误逐项记录并保留原始录制文件；
- `finally` 保证录制状态清理，避免残留“正在录制”。

## 测试策略

采用测试驱动开发，不依赖真实直播间或公网：

- `pathing`：名称清洗、根目录尾分隔符、主播/日期/标题组合、六种扩展名和分段模板；
- `ffmpeg_builder`：六种格式的分段与非分段命令、国内/海外参数、代理、headers、音频映射与容器；
- `process`：自然结束、注释停止、全局退出、启动异常、异常返回码、Windows `q` 和 POSIX `SIGINT`；
- `postprocess`：单文件及分段文件发现、TS 转 MP4、失败保留原文件、自定义脚本次数；
- `pipeline`：FFmpeg 与直连策略、仅音频不落入视频分支、结束原因与后处理门禁、状态清理；
- `main.py`：静态或窄接口接线测试，避免导入入口副作用；
- 回归：完整 `pytest`、Ruff 检查与格式检查、`py_compile`。

至少对 TS、FLV、MP4、MP3 执行完整管线模拟；对 MKV、M4A 验证命令与后处理。真实直播录制仅作为可选人工验证，不作为自动验收门禁。

## 验收标准

- `src/recorder/` 提供稳定且无 `main.py` 全局依赖的公共接口；
- `main.py` 不再包含六种格式的命令拼接和子进程生命周期实现；
- 六种格式的分段与非分段策略均有自动化测试；
- 猫耳/Look 音频重复录制、分段 FLV 模板、分段 M4A 容器和 MP4 转换条件四项缺陷均有回归测试并修复；
- 正常结束与用户注释停止采用一致的后处理规则；
- 直连 FLV 平台行为保持兼容；
- 完整测试、静态检查、格式检查和语法编译通过；
- 路线图准确记录 PR4 实施结果及无法自动覆盖的真实网络验证边界。
