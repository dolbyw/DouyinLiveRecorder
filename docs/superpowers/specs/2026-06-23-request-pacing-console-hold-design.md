# 监控请求均匀节流与 Windows 完成界面驻留设计

## 目标

解决两个已确认问题：

1. 多个直播间的监控请求在同一时刻成批发出，形成周期性请求尖峰。
2. Ctrl+C 完成转码后，主界面因控制台交互判断错误而立即退出。

本次不改变每个房间的目标监控周期、不限制同时录制数量，也不改变转码流程。

## 当前行为与根因

### 请求聚集

异步运行时为每个直播间同时创建一个 `RoomMonitor` 任务。每个任务立即执行第一次探测，再固定休眠配置的监控间隔。`AdjustableLimiter(3)` 只限制同时进入探测代码的任务数量，因此 14 个房间会以最多 3 个一批快速执行；完成后又以近似相同的周期再次唤醒，请求尖峰会长期重复。

兼容线程调度只在创建线程时使用“排队读取网址时间”，默认值可为 0；每个线程之后也按相似周期循环，因此同样可能保持同步。

### 完成界面自动关闭

真实退出链为：

```text
signal_handler
→ 等待录制与转码完成
→ phase = COMPLETE
→ wait_for_exit_key
→ sys.stdin.isatty() == False
→ 不读取按键，立即返回
→ sys.exit(0)
```

PyInstaller 配置虽然使用 `console=True`，但从当前 Windows Terminal 启动时，Python 标准输入包装器仍可能报告非 TTY。现有代码把该结果错误地等同于“没有控制台”，导致 `msvcrt.getwch()` 从未执行。

## 并发数量的准确语义

“同一时间访问网络的线程数”改为界面文案“探测并发”，默认值从 3 调整为 5。

它限制的是同时进行的直播状态获取、房间信息解析和播放地址探测任务，不限制：

- FFmpeg 录制进程数量
- 同时开播或正在录制的房间数量
- TS→MP4 转码任务数量
- 配置中的直播间总数

均匀节流控制“请求何时开始”，并发限制控制“慢请求最多重叠多少个”。采用节流后，正常网络下实际探测并发通常为 1；默认 5 用于容纳少量慢请求或超时请求，同时避免异常时无限堆积。

默认值变更覆盖：配置模型默认值、配置加载器缺省值、模板配置和本次新发行包中的现用配置。用户显式设置其他正整数时仍尊重用户配置。

## 全局均匀节流

### 节拍计算

每次加载有效房间列表后计算基础请求启动间距：

```text
基础间距 = 监控周期秒数 ÷ 有效异步监控房间数
```

例如 14 个房间、300 秒周期时，基础间距约为 21.43 秒。第一个请求可以立即开始，后续请求在全局节拍器中依次取得启动许可。

每个间距乘以 `0.9～1.1` 的随机系数，避免形成完全固定的机器节奏。随机扰动只影响请求启动时刻，不改变并发上限。

### 异步运行时

新增单一 `RequestPacer`，由全部 `RoomMonitor` 共享。探测顺序为：

1. 等待全局节拍许可。
2. 获取 `AdjustableLimiter` 并发许可。
3. 执行平台探测。
4. 释放并发许可。

节拍器使用单调时钟和异步锁。等待节拍时不占用并发许可，避免一个尚未开始请求的任务占住并发名额。

`RuntimeCoordinator` 每次刷新配置时，将监控周期和当前有效房间数传给节拍器。新增、删除或停用房间后，下一次配置刷新即调整基础间距。首次请求不人为等待完整间距。

### 兼容线程调度

对未进入统一异步运行时的平台，线程启动间距取以下较大值：

```text
max(用户配置的排队读取时间, 监控周期 ÷ 当前兼容房间数)
```

线程首次启动被均匀错开后，各线程仍按原监控周期独立循环；原有 ±5 秒循环扰动继续保留。该方式不新增第二套长期节拍器，也不会延迟已经开始的录制。

### 失败与重试

探测失败后的重试也必须重新经过全局节拍器，不能绕过节流形成错误风暴。任务取消和程序退出可立即中断节拍等待。

## Windows 完成界面驻留

### 控制台检测

Windows 改用 Win32 控制台输入句柄判断：

1. 调用 `GetStdHandle(STD_INPUT_HANDLE)`。
2. 排除 `NULL` 和 `INVALID_HANDLE_VALUE`。
3. 调用 `GetConsoleMode` 验证该句柄确实连接到控制台输入缓冲区。

只要控制台句柄有效，就调用 `msvcrt.getwch()` 等待一个按键，不再参考 `sys.stdin.isatty()`。

非 Windows 系统仍使用 `sys.stdin.isatty()` 和 `input()`。确实没有控制台句柄、句柄已关闭或读取发生 `EOFError/OSError` 时，记录原因并正常退出，避免后台任务永久挂起。

### 完成状态可见性

在进入按键等待之前：

1. 设置 `AppDisplayPhase.COMPLETE`。
2. 写入最终收尾事件。
3. 触发一次面板立即刷新，而不是等待后台面板线程的下一秒刷新。
4. 显示 `按任意键退出 | Ctrl+C 强制退出`。
5. 开始阻塞读取按键。

普通按键使主线程正常执行 `sys.exit(0)`。等待期间再次 Ctrl+C 继续调用现有 `ShutdownControl`，通过 `os._exit(130)` 立即退出。

## 组件边界

### `src/runtime/pacer.py`

实现与平台无关的异步请求节拍器，支持动态设置监控窗口与房间数。时钟、睡眠函数和随机函数均可注入，以便测试精确验证启动间距而不真实等待。

### `src/runtime/monitor.py`

只负责在探测前依次取得节拍和并发许可。录制、离线回调和成功回调保持现有顺序。

### `src/runtime/coordinator.py`

刷新配置时同步更新并发上限与节拍器房间数。协调器不直接休眠单个房间。

### `src/runtime/exit_wait.py`

封装 Win32 控制台句柄验证和单键读取。测试使用注入的句柄检查器和读取器，不依赖真实键盘。

### `main.py`

构建并共享一个异步节拍器；兼容房间按自动间距启动。完成收尾后先立即刷新最终快照，再等待按键。

## 界面变化

配置项“并发”改为“探测并发”，明确其作用范围。实时详情的下次检测倒计时继续以每个房间最近一次真实探测时间为基准；节拍排队造成的少量延后允许短暂显示 `00:00`。

不显示随机系数、内部等待队列或敏感直播源地址。

## 测试与验收

自动化测试覆盖：

1. 14 个房间、300 秒周期计算约 21.43 秒基础间距。
2. 第一个请求立即开始，后续请求按注入的随机系数依次等待。
3. 多个同时到达的任务不会越过节拍顺序。
4. 等待节拍时不占用并发许可。
5. 配置增删房间后动态更新间距。
6. 失败重试仍经过节拍器。
7. 用户显式并发值生效，缺省值为 5。
8. 兼容线程启动间距不小于监控周期除以兼容房间数。
9. Windows 标准输入报告非 TTY、但 Win32 控制台句柄有效时仍读取一个按键。
10. Windows 无控制台句柄时立即返回，不永久等待。
11. `COMPLETE` 最终画面在按键读取前被主动刷新。
12. 普通按键正常退出，第二次 Ctrl+C 的强制退出行为保持不变。
13. 全量测试、真实 FFmpeg 转码冒烟和 PyInstaller 打包继续通过。

人工验收使用 14 个房间、300 秒监控周期：观察日志中探测开始时间至少一个完整周期，请求应均匀分布而非三连发。随后让至少一个房间录制，按一次 Ctrl+C，确认转码完成界面持续保留；按普通键退出。再次运行并验证完成界面中的 Ctrl+C 可立即退出。

## 非目标

- 不绕过平台认证、验证码或其他风控机制。
- 不保证任何请求模式一定不会触发平台风控。
- 不限制录制并发或转码并发。
- 不增加用户必须理解的新节流配置项。
